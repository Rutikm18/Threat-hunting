⬡ Scenario 01 — Living-off-the-Land Lateral Movement via WMI + PowerShell

CRITICAL
T1047 · WMI T1059.001 · PowerShell T1021.006 · WinRM T1078 · Valid Accounts
⚠ Situation: Finance department workstation flagged by SIEM for high volume of WMI queries to multiple internal hosts at 2:30 AM. No alerts from AV/EDR. User account is an active service account with domain admin privileges.

Hunt Approach — Step by Step

  • 01
    Establish Baseline (What's Normal?) Query EDR for historical WMI activity from this host over last 90 days. Does it normally run wmic.exe? Does it normally initiate outbound WMI to remote hosts? Compare time-of-day activity patterns. Isolate the anomaly.
    // Splunk: WMI process creation from finance hosts
    index=endpoint host=finance-* EventCode=4688
    process_name="wmic.exe" OR "wmiprvse.exe"
    | stats count by host, user, process_name, CommandLine, _time
    | sort -_time
  • 02
    Map the Process Tree Pull full process ancestry in EDR. Is wmic.exe spawned by cmd.exe → PowerShell → WMI? Was PowerShell launched with encoded commands (-EncodedCommand, -Enc)? Check parent process. Encoded command = near-certain adversary technique.
    # CrowdStrike Falcon Query — suspicious PS ancestry
    event_simpleName=ProcessRollup2
    | where FileName="powershell.exe"
    AND CommandLine CONTAINS "-Enc"
    AND ParentBaseFileName="wmiprvse.exe"
    | table ComputerName, UserName, CommandLine, ParentCommandLine
  • 03
    Decode & Analyze Encoded PowerShell Extract the Base64 encoded blob from CommandLine. Decode it. Look for: download cradles (IEX, Invoke-WebRequest), AMSI bypass patterns, credential dumping (Mimikatz patterns), scheduled task creation, WMI persistence.
    # Python: decode encoded PowerShell
    import base64
    encoded = "JABjAGwAaQBlAG4AdA..."
    decoded = base64.b64decode(encoded).decode('utf-16-le')
    print(decoded)
    # Look for: IEX, Net.WebClient, DownloadString, AMSI bypass
  • 04
    Lateral Movement Confirmation Check Windows Event Log 4624 (logon) on destination hosts during the WMI activity window. Type 3 network logon from the service account? Cross-reference with WMI remote execution. Pull Sysmon Event ID 1 (process create) on destination hosts. Did wmiprvse.exe spawn anything?
  • 05
    Scope, Contain & Hand Off to IR Enumerate all hosts touched by the service account in the window. Build a timeline. Determine if credential dumping occurred (check for LSASS access: Sysmon Event ID 10). If confirmed compromise — isolate hosts, revoke/rotate the service account, initiate IR. Draft executive-ready summary in parallel.
◈ Key Hunt Metric: Dwell time from initial WMI execution to detection. If it's >24 hours, your detection coverage has a gap. Write a detection rule for WMI spawning PowerShell with encoded args immediately.

⬡ Scenario 02 — C2 Beaconing Hidden in HTTPS Traffic (Domain Fronting)

CRITICAL
T1071.001 · Web Protocols T1090.004 · Domain Fronting T1132 · Data Encoding T1568 · Dynamic Resolution
⚠ Situation: Darktrace flags a developer workstation making suspiciously regular HTTPS connections to *.azureedge.net every 60 seconds for 6 hours. No malware detected. Traffic volume is low — looks like legitimate CDN traffic at first glance.

Hunt Approach — Step by Step

  • 01
    Statistical Beaconing Analysis Pull all outbound HTTPS connections from the host over 24 hours. Calculate inter-arrival times between connections to each destination. True beaconing: low jitter (stddev <10s), regular intervals. Legitimate CDN: highly variable timing, larger volume.
    # Python — beacon detection via jitter analysis
    import pandas as pd, numpy as np
    df = pd.read_csv('netflow.csv')
    df['ts'] = pd.to_datetime(df['timestamp'])
    grp = df.groupby('dest_ip')
    beacons = grp['ts'].apply(
      lambda x: np.std(x.sort_values().diff().dt.seconds.dropna())
    )
    # Low std deviation = beacon candidate
    print(beacons[beacons < 5])
  • 02
    Inspect TLS Metadata (No Decryption Needed) Even without SSL inspection, JA3/JA3S fingerprints reveal the TLS client/server signature. Cobalt Strike, Metasploit, and custom C2s have known JA3 hashes. Check Zeek SSL logs. Also look at: certificate subject mismatch, certificate age (new = suspicious), SNI vs. actual destination IP mismatch = domain fronting.
    # Zeek SSL log analysis for JA3 fingerprints
    cat ssl.log | zeek-cut ts uid id.orig_h id.resp_h ja3 ja3s server_name
    | grep "azureedge.net"
    # Known Cobalt Strike JA3: 72a7c13d19b61af9aa2f70c0cdb3a79b
    # If SNI says *.azureedge.net but resolved IP is actor-controlled → fronting
  • 03
    DNS Resolution Audit Check DNS logs for the hostname being resolved. Does it resolve to Microsoft CDN IPs (legitimate) or to a different AS? Threat actors use domain fronting: SNI points to legit CDN, but Host header routes to actor's C2. Inspect HTTP Host headers if you have SSL inspection or proxy logs.
  • 04
    Endpoint Forensics — Find the Implant On the endpoint: check for unsigned DLLs loaded by legitimate processes (DLL side-loading is common). Look for persistence: Run keys, scheduled tasks, WMI subscriptions. Memory analysis (Volatility) to find injected shellcode in legitimate process memory space.
    # Volatility 3 — find injected memory regions
    python3 vol.py -f memory.dmp windows.malfind
    # Look for: RWX pages in non-image regions of legitimate processes
    # e.g., svchost.exe with injected shellcode → Cobalt Strike
  • 05
    Attribution & Containment Hash implant artifacts → VirusTotal / Malware Bazaar. Extract C2 config from memory (CS config parser). Identify campaign. Scope: which other hosts contacted same destination? Is this the patient zero or did it spread? Network-isolate host, preserve forensic image, escalate to IR.

⬡ Scenario 03 — Insider Threat: Data Exfiltration via Cloud Storage + DLP Bypass

HIGH
T1567.002 · Exfiltration to Cloud T1048 · Exfil Over Alt Protocol T1078 · Valid Accounts T1560 · Archive Collected Data
⚠ Situation: HR reports an employee gave 2 weeks notice yesterday. UEBA score for the user spiked 340% over baseline. Large file access volume on SharePoint, followed by traffic to a personal Google Drive account — not blocked by DLP because it's classified as "Google Workspace."
  • 01
    UEBA Investigation — What Changed? Pull the UEBA alert details. What behaviors drove the score? Typical insider threat signals: unusual off-hours access, accessing resources outside normal job scope, bulk file downloads, new USB device registration, email to personal accounts. Quantify the deviation from peer group baseline.
  • 02
    DLP & Cloud App Security (CASB) Review Check CASB logs (e.g., Microsoft Defender for Cloud Apps / Netskope). What was uploaded to Google Drive? File names, sizes, timestamps. Was it personal files or corporate data? Check SharePoint audit logs: which document libraries were accessed, what was downloaded, was it bulk download?
    // KQL: SharePoint mass download hunt — Microsoft Sentinel
    OfficeActivity
    | where Operation in ("FileDownloaded", "FileSyncDownloadedFull")
    | where UserId == "target@company.com"
    | where TimeGenerated > ago(14d)
    | summarize FileCount=count(), TotalBytes=sum(FileSizeBytes)
      by bin(TimeGenerated, 1h), UserId
    | where FileCount > 100
  • 03
    Network Telemetry — Exfiltration Volume Check proxy/firewall logs for upload traffic to Google Drive (drive.google.com / docs.google.com). Measure upload bytes. 500MB+ upload to personal cloud = significant. Check for other destinations: personal email (Gmail SMTP over 587), file sharing sites (WeTransfer, Mega.nz), Dropbox.
  • 04
    Legal & HR Coordination This is where technical and legal intersect. Do NOT alert the user. Loop in Legal and HR immediately. Document everything with timestamps. Preserve evidence forensically — legal hold on email, SharePoint, endpoint. Chain of custody matters. Your findings may go to court.
  • 05
    Remediation & Policy Gap Post-incident: why did DLP not catch Google Drive uploads? Because CASB was configured to allow "Google Workspace" broadly. Recommendation: content-aware DLP inspection for all cloud storage, block personal Google Drive accounts while allowing corporate Google Workspace, implement egress rate limiting alerts.

Hunt Hypotheses
◎ Every hunt starts with a falsifiable hypothesis. Format: "I believe [threat actor/technique] is occurring because [indicator/intelligence], detectable via [telemetry source/query approach]." Bad hunters search randomly. Good hunters test specific hypotheses and measure outcomes.
// HYPOTHESIS 01 — CREDENTIAL ACCESS
Adversaries are performing LSASS credential dumping via legitimate system utilities to evade AV detection on workstations in the engineering network.
Rationale: Recent Mandiant report documents TA505 using comsvcs.dll MiniDump technique. Engineering hosts have domain admin users frequently logged in, making them high-value targets. EDR rules focus on Mimikatz signatures, not LOLBin variants.
MITRE T1003.001 Sysmon EID 10 EDR Telemetry High Value
// Hunt: LSASS access via comsvcs MiniDump
event_simpleName=ProcessRollup2
| where CommandLine MATCHES ".*comsvcs.*MiniDump.*lsass.*"
OR CommandLine MATCHES ".*Out-Minidump.*"
OR (FileName="rundll32.exe" AND CommandLine CONTAINS "comsvcs")
| table ComputerName, UserName, CommandLine, SHA256HashData
// HYPOTHESIS 02 — PERSISTENCE
A threat actor has established WMI event subscription persistence on servers in the DMZ following last quarter's vulnerability exposure window.
Rationale: WMI subscriptions are rarely used legitimately on servers, persist across reboots without touching disk in traditional locations, and are invisible to most AV products. The DMZ had unpatched systems for ~3 weeks — a likely dwell window.
MITRE T1546.003 WMI Repository Sysmon EID 19/20/21
# PowerShell: enumerate WMI event subscriptions
Get-WMIObject -Namespace root\subscription -Class __EventFilter
Get-WMIObject -Namespace root\subscription -Class __EventConsumer
Get-WMIObject -Namespace root\subscription -Class __FilterToConsumerBinding
# Legitimate: empty or known SCCM/management entries only
# Suspicious: any entry with script consumer / command line consumer
// HYPOTHESIS 03 — DISCOVERY & RECONNAISSANCE
An attacker with initial access is performing internal Active Directory reconnaissance using built-in Windows tools to avoid triggering endpoint detections.
Rationale: LOLBin AD recon (nltest, net group, dsquery, ldapsearch) is common post-initial-access behavior. With broad initial access from phishing campaigns circulating targeting our sector this month, this is a high-probability scenario to hunt against immediately.
MITRE T1018 · T1069 · T1087 Windows Event 4688 EDR Process Telemetry
// Splunk: AD recon via LOLBins
index=endpoint EventCode=4688
process_name IN ("nltest.exe","net.exe","dsquery.exe","whoami.exe","ipconfig.exe")
| bin _time span=5m
| stats dc(process_name) as unique_recon_tools, values(CommandLine) by host, user, _time
| where unique_recon_tools >= 3
// 3+ recon tools from same user in 5min = strong signal
// HYPOTHESIS 04 — C2 COMMUNICATION
A compromised endpoint is communicating with a C2 server over DNS tunneling, bypassing HTTP-based proxy controls and DLP.
Rationale: DNS egress is rarely restricted. DNS tunneling tools (dnscat2, iodine) encode C2 traffic in DNS queries. Organizations with strong HTTP/S controls are increasingly targeted via DNS. Recent threat intel shows UNC3524 using DNS-based C2 against financial targets.
MITRE T1071.004 DNS Logs Zeek DNS Evasion
# Hunt: DNS tunneling detection — high entropy subdomains
import math, collections
def entropy(s):
  p = [s.count(c)/len(s) for c in set(s)]
  return -sum(x*math.log2(x) for x in p)
# Flag subdomains with entropy > 3.5 and length > 30 chars
# Also flag: unusually high query rate to single domain, TXT/NULL record queries
// HYPOTHESIS 05 — SUPPLY CHAIN / TRUSTED TOOL ABUSE
A legitimate software update mechanism (e.g., software deployment tool) is being abused by an attacker who has compromised an internal build server to push malicious payloads to endpoints.
Rationale: Supply chain attacks (à la SolarWinds, 3CX) leverage trusted software update paths. Internal build servers with write access to deployment systems are high-value lateral movement targets. If a developer workstation was compromised, build server access is the next pivot.
MITRE T1195.002 Supply Chain Code Signing Build Integrity

Senior Interview Q&A
◈ These are real questions asked at senior / staff threat hunter interviews at FAANG security teams, investment banks, MSSPs, and government agencies. Click each question to reveal the expert-level answer.
Q01 You're told to build a threat hunting program from scratch at a company with only a basic SIEM and no EDR. Walk me through your first 90 days. ▾
// EXPERT ANSWER

Days 1–30: Visibility Assessment. You can't hunt what you can't see. Audit your log sources: what's in the SIEM? Windows events (4624, 4625, 4688, 4698, 4720, 4732)? DNS? Proxy? Netflow? Firewall? Map your visibility against MITRE ATT&CK — you'll immediately see massive blind spots. This becomes your gap report.

Days 31–60: Quick Wins & Intel Integration. Without EDR, focus on network-based hunting (DNS anomalies, beaconing via netflow, unusual outbound connections) and Windows event log hunting (LOLBin usage, abnormal service installs, bulk account lockouts). Integrate a free threat intel feed (MISP, AlienVault OTX) against your SIEM. These quick wins build stakeholder trust and justify EDR budget.

Days 61–90: Framework & Metrics. Define your hunt workflow: hypothesis → execution → finding → detection or false positive. Set baseline metrics: hunts per month, TTD (time to detect), dwell time estimates, detection coverage percentage. Present findings to leadership with ROI framing. Begin the EDR procurement argument with evidence you've gathered.

Key principle: Don't try to build the perfect program. Build a program that improves incrementally, measures itself, and demonstrates value. Tools follow process, not the other way around.

Q02 Explain the difference between IOC-based hunting and TTP-based hunting, and why a mature program prioritizes TTPs. ▾
// EXPERT ANSWER

IOC-based hunting searches for specific artifacts: IP addresses, file hashes, domain names, registry keys. It's fast and operationally useful, but fundamentally reactive and brittle. Adversaries rotate IOCs in hours — a C2 IP blocked today is abandoned within 24 hours. IOCs are at the bottom of the Pyramid of Pain (David Bianco's model) — easy for attackers to change, limited long-term value.

TTP-based hunting targets adversary behaviors: how they execute code, persist, move laterally, exfiltrate. These are at the top of the Pyramid of Pain — extremely costly for adversaries to change because it requires rewriting tooling, retraining operators, or abandoning entire attack chains. A hunt for "WMI spawning PowerShell with encoded commands" catches Cobalt Strike, Metasploit, and custom implants regardless of their specific IOCs.

Mature programs use both strategically: IOCs for fast, automated enrichment and blocking at scale; TTPs for proactive hunting, detection engineering, and measuring adversary behavior across campaigns. A common interview mistake is treating them as mutually exclusive — they're complementary layers.

Practical example: When SolarWinds hit, organizations with only IOC-based defenses were blind. Organizations hunting TTPs for "unexpected network connections from build/monitoring tools" or "signed binaries making unusual DNS requests" had a detection path even before IOCs were published.

Q03 A process on a host is making thousands of LDAP queries to the domain controller. How do you determine if this is normal Active Directory operation or adversarial reconnaissance? ▾
// EXPERT ANSWER

Start with context, not conclusions. LDAP queries to DCs are normal — AD health, login validation, group policy, monitoring tools all generate them. The question is whether the pattern is anomalous for this specific host and process.

Key differentiators to investigate: (1) Process identity — is it a known enterprise tool (SCCM, Okta sync, monitoring agent) or an unexpected binary? (2) Query content — benign queries are usually narrow-scope; recon queries enumerate entire AD: all users, all groups, adminSDHolder, unconstrained delegation targets, SPNs (Kerberoasting prep). (3) Rate and timing — bulk enumeration in seconds vs. normal trickle over hours. (4) Account context — is a non-privileged user account making admin-scope queries?

Technical investigation: Enable LDAP diagnostic logging on DCs (Event ID 1644 with appropriate filter). Review the search filters in queries. Bloodhound/SharpHound LDAP queries have characteristic patterns: querying for ms-DS-AllowedToDelegateTo, adminCount=1, or requesting all user objects with specific attributes in bulk. Compare to your LDAP baseline if you have one.

If adversarial: This indicates post-exploitation — attacker has initial access and is mapping the AD environment to plan lateral movement and privilege escalation. Scope the initial access, contain the host, initiate IR. Check if any Kerberoastable accounts were identified (they'll be targeted for ticket requests next).

Q04 How would you detect a Kerberoasting attack in progress, and what's your response if you find it? ▾
// EXPERT ANSWER

What is it: Kerberoasting requests Kerberos TGS tickets for service accounts with SPNs. These tickets are encrypted with the service account's NTLM hash, extracted offline, and cracked. No elevated privileges needed for the request — any domain user can do it.

Detection via Windows Event ID 4769 (Kerberos Service Ticket Request). Signals: (1) RC4 encryption type (0x17) requested instead of AES — RC4 is weak, chosen specifically for faster cracking; (2) Multiple TGS requests in short timeframe from one account; (3) Requests targeting service accounts outside of business hours or from a workstation that doesn't normally use those services; (4) Requesting tickets for high-privilege service accounts (backup, SQL, admin) from a non-server host.

// KQL: Detect Kerberoasting — bulk RC4 TGS requests
SecurityEvent
| where EventID == 4769
| where TicketEncryptionType == "0x17" // RC4
| where ServiceName !endswith "$" // exclude computer accounts
| summarize count() by bin(TimeGenerated,5m), Account, IpAddress
| where count_ > 5 // 5+ service accounts in 5min = Kerberoasting

Response: Identify which service accounts had tickets requested. Immediately assess their password strength and privilege level. If an account hasn't had its password rotated in >1 year, assume it's been cracked. Rotate all affected service account passwords immediately. Longer term: convert service accounts to Managed Service Accounts (MSAs) which have auto-rotating 240-char passwords that are uncrackable. Implement honeypot SPNs (fake Kerberoastable accounts that alert immediately when a ticket is requested — no legitimate user should ever request them).

Q05 You discover an attacker has been in your environment for 47 days. How do you determine scope without tipping them off? ▾
// EXPERT ANSWER

First principle: observe before acting. Do not block, isolate, or alert the attacker until you've fully mapped their footprint. Premature containment causes them to destroy evidence, accelerate exfiltration, or deploy ransomware immediately.

Timeline reconstruction: Identify patient zero — earliest artifact of compromise. Work forward chronologically. Use EDR telemetry, SIEM logs, authentication logs, DNS/proxy logs, email logs. Build a minute-by-minute timeline. Tools: Timeline Explorer, Plaso/log2timeline, custom Python. Answer: When did they arrive? Via what vector? What have they touched?

Scope mapping: From patient zero, trace all lateral movement. Every remote authentication, every WMI/PSRemoting/RDP session, every scheduled task creation, every service install. Map the blast radius on a network topology diagram. Identify: data accessed, credentials obtained, persistence mechanisms placed, potential exfiltration.

Covert monitoring: If resources allow, set up network TAP / mirror on key segments to capture live attacker activity. Don't block C2 yet — watch what they're doing. Are they collecting data? Moving toward crown jewels? This intelligence shapes your containment strategy. Coordinate with Legal — covert monitoring of an intruder in your own network is generally lawful but verify jurisdiction.

Containment strategy: When ready, execute coordinated simultaneous containment across ALL identified compromised hosts at once — not host by host. Simultaneous is critical. Sequentially isolating hosts telegraphs your awareness and gives the attacker time to destroy evidence and pivot elsewhere. Have IR team and comms plan ready before you pull the trigger.

Q06 How do you measure the effectiveness of a threat hunting program? What metrics do you report to leadership? ▾
// EXPERT ANSWER

Technical metrics (for the team): Mean Dwell Time Reduction (how long adversaries existed before detection — trend over quarters), Hunt-to-Detection Conversion Rate (% of hunts that generate net-new detections), MITRE ATT&CK Coverage Score (what % of relevant TTPs do your detections cover), False Positive Rate per detection rule, Number of confirmed incidents discovered through hunting vs. reactive alerts.

Executive metrics (for leadership): Risk reduction narrative is more powerful than raw numbers. "Our hunting program identified 3 incidents this quarter that evaded all automated controls — estimated average dwell time if undiscovered would have been 40+ days." Translate technical findings to business risk: "Adversary had access to IP repository for 12 days." Cost avoidance: "Identifying this before ransomware deployment avoided estimated $2–8M incident response and recovery cost."

Trap to avoid: Don't measure success by volume of hunts run. A hunter running 50 low-quality hunts with zero findings is less valuable than one running 10 rigorous hypothesis-driven hunts that discover 2 true positives. Quality over quantity. Measure outcomes, not activity.

Maturity model reporting: Use a hunting maturity model (HM0–HM4) to track program evolution. Show leadership that you're progressing from reactive (HM0) to data-driven automated hypothesis generation (HM4). This justifies investment and demonstrates strategic thinking.

◎ Senior Hunter Mindset: Interviewers at this level aren't looking for correct answers — they're looking for how you think. Walk through your reasoning out loud. Show your methodology. Acknowledge uncertainty and explain how you'd resolve it. Demonstrate attacker empathy — you understand adversary goals, not just their tools.
